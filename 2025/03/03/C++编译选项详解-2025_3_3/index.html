C.编译选项介绍

众所周知，Dev-c++ 是可以配置编译选项的，考试也会指定编译选项，接下来对 OI 常用编译选项进行简单介绍。

然后我们接下来所介绍的编译选项，可以选择在 exe 后加上。（其实位置没有要求）。

-std=c++14 -std=c++17 -std=c++98 可以用于指定 c++ 版本，注意你的编译器必须支持。

-Wall -Wextra 可以开启更多警告，更能够是你发现一些小错误。

-Wl,-stack=12345678 可以用于限定栈空间，防止递归爆栈（现在竞赛栈空间都默认和内存限制一致，所以最好开上），后面数字的单位是字节 Byte。注意使用这个时中间有逗号，且二者是整体，不要分开。

-O0 -O1 -O2 -O3 -Ofast 可以调节编译优化。-O0 表示不进行优化，-O1 表示只进行少量优化，-O2 表示进行相当程度的优化（竞赛强制使用），-O3 表示进行更为激进的优化，以及最为激进的 -Ofast。注意以上优化不会出错的前提是你的代码没有 UB（未定义行为，如有符号类型溢出）。

-Dxxxx 可以用于在编译时定义一个名为 xxxx 的宏。作用是什么呢？你可以配合 #ifdef 和 #endif 来做很多有意思的事，比如这份代码：

-fsanitize=address,undefined 可以用于检测代码运行时的 UB，如果出现 UB 会直接 RE，注意使用此编译选项需要编译器支持，同时会严重拖慢代码运行速度。

D.关于 NOI Linux 中 IDE 的评价

VS code（自己使用强推）

优点：强大的代码补全，如果可以安装扩展功能非常强大，自带终端。

缺点：扩展和中文翻译需要联网下载扩展，比赛别想用。编译需要手写 json 文件，略显麻烦。

Sublime Text（比赛和平时都可以使用）

优点：比 VS code 更轻量化，有还算可以的代码补全，有中文，可以单文件编译，性能好，可以打开几百行大样例。

缺点：终端不显示返回值，修改编译选项需要手写 json 文件（可以自己上网搜模板），不支持调试。

这里提供一个我自己用的：

cmdRun 可以在 windows 下打开终端输入，Run 则使用文件读写。

注意我使用了硬编码文件路径，因为我需要把 exe 和 cpp 分开至不同文件夹。你可以进行修改。

Geany（比赛强推）

优点：轻量化的 IDE，支持单文件编译，有中文，修改编译选项可以直接从菜单修改，开箱即用，基本不需要配置。

缺点：不支持调试，代码补全相当鸡肋。

Code::Blocks

优点：唯一支持调试的，有代码补全。

缺点：过于笨重，无中文，修改编译选项也比较麻烦。

Ex.P3275 [SCOI2011] 糖果

本来是差分约束的模板题，解决本题的关键即在于，每个关系都可以构成图上的边。

借助差分约束的思想，因为求的是最小值，所以应当跑最长路，因此每个关系可以这样转化：

  • 如果 X=1X=1X=1，表示第 AAA 个小朋友分到的糖果必须和第 BBB 个小朋友分到的糖果一样多，相当于 AAA 和 BBB 连一条双向边。
  • 如果 X=2X=2X=2，表示第 AAA 个小朋友分到的糖果必须少于第 BBB 个小朋友分到的糖果，因为 A<B⇒A+1≤BA < B \Rightarrow A+1 \le BA<B⇒A+1≤B，所以我们可以连一条从 AAA 向 BBB 的权值为 111 的单向边。
  • 如果 X=3X=3X=3，表示第 AAA 个小朋友分到的糖果必须不少于第 BBB 个小朋友分到的糖果，因为 A≥BA \ge BA≥B，所以我们可以连一条从 BBB 向 AAA 的权值为 000 的单向边。
  • X=4,5X = 4,5X=4,5 同上。

然后我们就可以建立超级源点，跑 spfa 求最长路。

所以做完了吗？很不幸，这么做理论最多 30 pts\text{30 pts}30 pts，spfa O(NK)\Omicron(NK)O(NK) 的复杂度显而易见不能通过本题，已经被 hack 数据卡死。而且绝对不要使用任何讨论区提到的各种不靠谱的优化，除了堆优化 spfa 在非负权图上有正确的复杂度（但是将不可以跑最长路，同时负权图可以卡到指数级），剩下的 spfa 都可以通过构造数据卡飞。

关于 SPFA，它已经死了。

因此，我们需要想别的方法。可能有同学已经想到了，条件 1,3,51,3,51,3,5 所连出的边是可以进行缩点的，然后再连 2,42,42,4 的边，若最后的图是一个 DAG，那么显然有解，跑一遍 topo+dp 求最长路即可。若最后的图有环，则说明出现冲突，无解。
